## 🔴 严重问题（2 条，均已实测复现）

### R1 — command_pose 全零检测漏洞（最关键）

`quaternion_angle_diff` 对全零四元数会除以 0 范数，产生 `NaN`。而 **`NaN > threshold` 在 Python 里恒为 `False`**，所以锁定部件全零的数据会**假通过**帧差异检查。

- **实测复现**：构造了一个 `arm_left` 全零的 command merge_pose 文件，`verify_data()` 无异常直接通过。
- **影响**：这正是背景里那个 recorder 侧 fallback 0.0 bug（`mcap_to_hdf5.cpp:995-1000`）应被质检层拦住的场景，但实际拦不住 —— 废数据（action 被锁部件全零）会静默流到训练集。
- **修复建议**：四元数在算 angle_diff 前先检查范数，范数 ≈ 0（低于阈值如 1e-6）→ 直接判 `fail`，而不是让 NaN 蒙混过关。同时对整段 command_pose 加"全零/常零段"专项检测。

### R2 — 时间戳校验性能坑（约 2400 倍差距）

`_check_mismatch_timestamp` 逐帧逐 key 用 `hdf5_obj[key][idx]` 做 HDF5 单元素读取。

- **实测复现**：20000 帧 × 4 key 耗时 **3.376s**；向量化后 **0.0014s**，约 **2400 倍**差距。
- **影响**：真实数据 key 更多、帧更多，批量质检会被严重拖慢，成为吞吐瓶颈。
- **修复建议**：一次性把 dataset 读进内存 numpy 数组再做向量化比较，禁止逐元素索引 HDF5。

## 🟡 改进项（7 条，其中 3 个已实测复现）

## ⚠️ 一处需你确认（映射规则一致性）

时间戳对齐检查的语义存在代码 vs 文档不一致：
- **代码实现**：比较的是"各 timestamp key **互相对齐**"
- **《HDF5 → LeRobot v3 映射规则》第 3 节**：写的是"**以 time 为基准**，各 `*_timestamp` 与 time 最大帧级偏差 > 0.2s → quality=fail"

这两个语义不一样（互相对齐 vs 以单一基准对齐）。需要你确认哪个是预期行为，再统一代码和文档。
